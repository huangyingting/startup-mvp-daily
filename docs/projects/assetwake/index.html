为什么值得做

这件事现在值得做，是因为两条曲线正在第一次真正交叉：一边是 2024 到 2025 年创业公司关闭与解散数量持续上升，形成了稳定的“数字遗产供给”；另一边是 AI 模型训练、强化学习和企业 AI 开发，对真实代码库、真实流程文档和真实协作语料的需求迅速增加。SimpleClosure 在新闻里给出的信号很强：需求侧合同季度增长 150%，被授权的数据集数量增长 50%，而且项目已经分析了约 30 亿 token、接近 100 家参与公司。这说明市场不再满足于公开 GitHub 仓库或纯合成数据，买方开始追求带着真实产品演进痕迹、真实工程决策和真实业务上下文的私有资产。与此同时，绝大多数倒闭创业公司并没有能力在极短时间里完成资产盘点、PII 清理、IP 权属确认和买家对接，于是“本来可以回收的钱”会因为流程太乱而直接蒸发。

AssetWake 的切口不是做一个冷冰冰的拍卖站，也不是把停业公司当作法务 checklist，而是把“关停”重新定义成一个可以被产品化的资产变现流程。它把最容易失控的几个环节——资产识别、可售性判断、敏感信息清理、买家匹配、交易留痕和收益分配——挂在同一条 exit timeline 上。这样产品的价值不只是“帮你找买家”，而是帮各方在最混乱、最情绪化、最缺时间的阶段里，把还剩下的数字资产真正转成可验证的回收金额。

Problem

要解决什么问题？

今天多数创业公司在关停时，真正流失的往往不是办公家具，而是散落在 Git 仓库、Slack、Notion、Google Drive、客户实施文档和内部 runbook 里的知识资产。问题在于，这些资产虽然有潜在价值，却很少有人能快速回答四个关键问题：第一，这个东西法律上能不能卖；第二，里面有没有客户数据、员工隐私、保密材料或第三方授权限制；第三，应该按源码、数据集、流程包还是 IP 组合来定价；第四，成交后钱该如何在创始团队、投资人和债权结构之间分配。关停阶段通常时间短、资源少、情绪重，创始人更关心尽快结束，投资人也缺少统一工具去判断什么值得回收。结果就是大量本可用于 AI 训练、产品加速或企业知识复用的资产，最终要么被粗暴删除，要么永远躺在没人再打开的工作区里。

目标用户

  • 准备关闭公司的 pre-seed 到 Series B 创始人团队，需要在有限时间内判断哪些数字资产还能回收价值、哪些必须清理删除
  • 管理投资组合关停、希望把“全部归零”改成“至少回收一部分数字资产价值”的种子基金、早期 VC 与 venture studio 团队
  • 为创业公司处理解散、清算、财务收尾与法务交接的律师事务所、fractional CFO 与 wind-down 顾问团队

MVP

这个原型包含什么

如果一个轻量的 asset recovery cockpit 能在创业公司进入 wind-down 阶段后，快速接入代码库与工作区数据，自动完成资产分类、敏感信息扫描、可售性分层、买家包生成和收益分配测算，那么创始人、投资人和清算顾问会愿意先把它加在现有关停流程之上，因为它直接替代的是最难、最乱、最容易错失价值的那段数字资产处置工作。

资产盘点雷达

接入 GitHub、GitLab、Slack、Notion 与 Google Drive 后，自动识别源码、文档、数据导出、流程模板与品牌资产，并按“可授权 / 需审查 / 必须删除”三层给出优先级，帮助团队先看清还有哪些东西值得救。

权属与脱敏流水线

对代码库和工作区材料执行 PII、客户标识、密钥、受限合同字段与第三方许可证扫描；同时把创始人、员工、客户协议和投资文件映射成权属检查清单，避免把不能卖的东西误打包进交易包。

买家包生成器

把可售资产自动整理成不同买家可理解的包：训练数据包、工程代码包、运营流程包或 IP 组合包，并附上资产摘要、行业标签、技术栈、可用范围与样例预览，减少每次从零解释的成本。

回收收益分配板

在交易接近完成时，系统按清算顺序、投资条款和约定成本生成收益分配预估，帮助创始人、投资人和顾问提前看到“如果按这个价格成交，最后各方大概拿多少”。

Use case

SaaS 团队关停前的最后 14 天

最常见的损失，不是卖不出去，而是根本来不及知道该卖什么。

  • 创始团队在决定关闭产品后，将代码库、工作区和历史交付资料接入 AssetWake，系统自动生成按资产类型排序的回收清单
  • 平台对每一类资产执行脱敏和权属扫描，标出涉及客户数据、第三方授权或合同限制的高风险内容，并建议是删除、隔离还是人工复核
  • 团队据此优先生成可出售的源码包与流程包，在关闭基础设施之前完成首轮买家投放和意向收集
北极星指标：从接入到首个可出售资产包生成的天数 / 被成功保留下来的高价值资产比例 / 关停前完成挂牌的资产数量

Use case

投资组合的批量资产回收

基金最怕的不是失败本身，而是每一家失败公司都用不同方式把剩余价值悄悄蒸发掉。

  • VC 平台团队把多个进入 wind-down 的项目拉入同一个投资组合视图，系统统一展示每家公司可回收的数字资产类型与预计处理难度
  • AssetWake 自动对资产做标准化标签，方便基金区分哪些更适合卖给 AI labs、哪些适合卖给并购方或行业软件公司
  • 平台团队在同一界面跟踪买家兴趣、清理进度和预估回收金额，把“失败项目复盘”升级成“失败项目回收运营”
北极星指标：单家关停项目的平均回收金额 / 平均资产处置周期 / 投资组合层面的数字资产回收率

Use case

律师与顾问的合规退出包

很多交易不是没人要，而是顾问不敢确认这包材料到底能不能安全交出去。

  • wind-down 顾问在平台里查看每份交易包的脱敏日志、权属说明与风险标记，快速判断是否还需要补充协议或人工复核
  • 系统生成一份 buyer-ready 的审计摘要，说明哪些内容已移除、哪些权限受限、哪些内容仅可授权不可转让
  • 交易完成后，平台保留完整的操作时间轴和交付证明，方便后续争议处理、LP 汇报和内部留档
北极星指标：顾问人工审查小时数 / 因合规不清导致的交易中断率 / 交付后争议发生率

Stack

当前技术栈

  • 第一阶段先用静态展示页验证“创业失败后的数字资产回收”这条叙事是否能打动创始人、VC 平台团队与 wind-down 顾问，再收集 demo 申请。
  • 产品端优先做 Web 控制台，不替代原有法务或文件存储工具，而是围绕 asset、rights check、redaction job、buyer package 和 recovery waterfall 五个核心对象展开。
  • 后端可用 Postgres 或 Supabase 存储资产元数据、扫描结果、买家兴趣与收益分配记录，再配合对象存储保存脱敏后的可交付包。
  • AI 层重点不是生成内容，而是做资产分类、可售性摘要、敏感字段识别和买家包自动说明，把混乱材料压缩成可交易对象。
  • 集成层优先连接 GitHub/GitLab、Google Drive、Notion、Slack 导出和数据脱敏服务，让 AssetWake 成为现有关停流程上的操作面，而不是新的数据孤岛。
  • 所有推荐都必须 evidence-first：任何“可出售”判断都要附带风险来源、脱敏动作和人工复核建议，避免产品被视为会制造法律风险的黑盒。

Risks

主要风险

  • 如果权属判断和脱敏能力不够稳，产品一次误售敏感资料就可能造成不可逆的法律与声誉风险。
  • 不同公司合同结构、客户协议和第三方许可证差异巨大，早期若试图完全自动化，容易在复杂 case 上失真。
  • 买家流动性可能高度不均衡，若平台只会整理资产、不能带来真实需求，就会退化成漂亮但无法变现的档案工具。
  • 创始人在关停阶段通常情绪和时间都很紧张，任何部署过重或解释成本太高的产品都可能被直接放弃。
  • 围绕 AI 训练数据的政策、隐私和版权规则仍在变化，产品需要持续更新授权模板与合规边界。

Next steps

如果继续迭代，接下来做什么

  • 先聚焦最标准化、最容易成交的源码资产，验证不同买家究竟更看重测试覆盖、部署脚本、行业上下文还是真实产品演进历史。
  • 把权属与脱敏流水线做成模板库，优先覆盖 SaaS、fintech 与 healthcare 三类最常见、同时合规要求最高的创业公司资产。
  • 与专门处理解散与清算的律师、venture studio 和基金平台团队建立渠道合作，让 AssetWake 成为 wind-down 的默认数字资产回收入口。
  • 继续向交易后环节延伸，加入收入分配对账、交付凭证和后续授权管理，形成完整的 startup afterlife operating system。
